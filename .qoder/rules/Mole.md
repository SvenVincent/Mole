# Mole 项目开发规范

## 开发流程

### 1. 需求分析与调研阶段
- 深入理解项目需求和目标
- 调研相关技术方案和最佳实践
- 分析潜在的技术挑战和风险
- 确定技术栈和架构方向

### 2. 任务规划与拆分
- 将大型任务拆分为多个独立的小任务
- 每个小任务应该：
  - 目标明确，可独立完成
  - 工作量控制在合理范围（建议1-3天）
  - 有清晰的验收标准
  - 依赖关系明确
- 制定任务优先级和时间计划

### 3. 文档编写
在开发前完成以下文档：
- **技术设计文档**：架构设计、模块划分、接口定义
- **开发计划**：任务列表、时间安排、里程碑
- **API文档**：接口规范、数据结构定义
- **测试计划**：测试策略、测试用例

### 4. 迭代开发
按照任务拆分结果，逐步推进：
- **Step 1**: 完成第一个小任务
  - 编写代码
  - 单元测试
  - 代码审查
  - 提交代码
- **Step 2**: 完成第二个小任务
  - 重复上述流程
- **Step N**: 依次完成所有小任务

### 5. 代码规范
- 遵循项目的代码风格指南
- 保持代码简洁、可读性强
- **添加必要的注释和文档**
  - 为所有公共接口添加注释，说明功能、参数、返回值
  - 为复杂逻辑添加行内注释，解释实现思路
  - 为关键算法添加注释，说明时间/空间复杂度
  - 使用统一的注释风格（如 Doxygen、JSDoc 等）
  - 注释应当解释"为什么"而非"是什么"
- 避免代码重复，提取公共逻辑

### 6. 测试要求
- 每个小任务完成后进行单元测试
- 集成测试确保模块间协作正常
- 回归测试确保不影响已有功能
- 编写测试用例文档

### 7. 版本控制
- 使用 Git 进行版本管理
- 提交信息清晰明确
- 小步提交，频繁集成
- 使用分支管理不同的功能开发

### 8. 持续改进
- 定期回顾开发过程
- 总结经验教训
- 优化开发流程
- 更新文档和规范

## 示例工作流

1. 需求调研 → 2. 拆分任务 → 3. 编写文档 → 4. 开发 Task-1 → 5. 测试 Task-1 → 
6. 开发 Task-2 → 7. 测试 Task-2 → ... → N. 集成测试 → N+1. 发布

## 注意事项
- 不要跳过调研和规划阶段
- 文档先行，代码跟随
- 小步快跑，持续集成
- 重视代码质量和测试覆盖率
- **养成良好的代码注释习惯，让代码自解释且易于维护**
